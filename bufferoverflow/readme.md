# bufferoverflow

버퍼 오버플로우가 어떻게 프로그램의 실행 흐름을 바꾸는지 보여준다.


## 배경 지식 - 스택의 구조

함수를 호출하면 메모리의 "스택" 영역에 다음 정보가 나란히 쌓인다:

```
[높은 주소]
+---------------------------+
| 리턴 주소                  |   <- 함수가 끝나면 여기 적힌 주소로 돌아간다
+---------------------------+
| 지역 변수들                |   <- 함수 안에서 선언한 변수들
+---------------------------+
[낮은 주소]
```

데이터를 쓸 때 낮은 주소 -> 높은 주소 방향으로 채워진다.
버퍼 크기를 넘겨서 쓰면 그 뒤에 있는 리턴 주소를 덮어쓸 수 있다.


## 시나리오

로그인 시스템이 파일에서 사용자 이름을 읽어와서 내부 버퍼에 복사한다.
실제 스택 공격은 OS 보호 기능 때문에 복잡하므로,
같은 원리를 구조체로 시뮬레이션한다:

```cpp
struct LoginContext {
    char username[8];            // 지역 변수 역할 (8바이트 버퍼)
    bool (*authCheck)(...);      // 리턴 주소 역할 (함수 포인터)
};
```

`strcpy`는 입력 길이를 체크하지 않는다.
입력이 8바이트를 넘으면 `username`을 넘어서 `authCheck`까지 덮어쓴다.


## 테스트 흐름

### 1단계: 정상 입력 (admin)

`input_admin.bin`에서 "admin"(5바이트)을 읽어온다.
버퍼(8바이트)에 충분히 들어가므로 오버플로우 없음.
`StrictAuthCheck`가 "admin"을 확인하고 접근을 허용한다.

### 2단계: 정상 입력 (guest)

`input_guest.bin`에서 "guest"(5바이트)를 읽어온다.
역시 버퍼 범위 내이므로 오버플로우 없음.
`StrictAuthCheck`가 "admin"이 아니므로 접근을 거부한다.

### 3단계: 공격 입력

`input_attack.bin`에서 공격자가 조작한 데이터를 읽어온다.

공격 파일의 내용:
```
[AAAAAAAA][AlwaysAllow 함수의 주소]
 8바이트        8바이트
```

프로그램은 이것을 정상적인 이름이라고 생각하고 `strcpy`로 복사한다.
8바이트를 넘기 때문에 `authCheck`가 `AlwaysAllow`로 덮어써진다.
결과적으로 누가 로그인하든 무조건 허용된다.


## 실행 결과 예시

```
=== 1단계: 정상 파일로 로그인 (admin) ===

  파일에서 읽은 값: "admin" (5바이트)
  버퍼 크기: 8바이트
  authCheck 변조: 없음
  [strictAuthCheck] 'admin' 권한을 확인합니다...
  [strictAuthCheck] 관리자 확인됨. 접근 허용.
  결과: 접근 허용

=== 2단계: 정상 파일로 로그인 (guest) ===

  파일에서 읽은 값: "guest" (5바이트)
  버퍼 크기: 8바이트
  authCheck 변조: 없음
  [strictAuthCheck] 'guest' 권한을 확인합니다...
  [strictAuthCheck] 관리자가 아닙니다. 접근 거부.
  결과: 접근 거부

=== 3단계: 공격자가 조작한 파일로 로그인 ===

  공격 전 authCheck: 00007FF658FD1060 (strictAuthCheck)
  파일에서 읽은 크기: 17바이트 (버퍼 8바이트를 초과!)
  버퍼 크기: 8바이트
  공격 후 authCheck: 00007FF658FD1070 (alwaysAllow = 00007FF658FD1070)
  authCheck 변조: 발생!
  [alwaysAllow] *** 권한 체크 우회! 무조건 허용! ***
  결과: 접근 허용
```


## 핵심 포인트

- `strcpy` 같은 길이 체크 없는 함수가 왜 위험한지 보여준다.
- 버퍼 오버플로우로 함수 포인터(실제로는 리턴 주소)를 덮어쓸 수 있다.
- 외부 입력(파일)이 프로그램 내부의 실행 흐름을 바꿀 수 있다.

## 이 데모와 실제 공격의 대응 관계

| 이 데모에서 | 실제 공격에서 |
|-------------|---------------|
| username[8] | 함수의 지역 변수 (스택) |
| authCheck 함수포인터 | 스택의 리턴 주소 |
| strcpy(ctx->username) | strcpy, gets 등 취약한 함수 |
| AlwaysAllow | 공격자의 악성 코드 (shellcode) |
| input_attack.bin | 조작된 네트워크 패킷, 파일 등 |
