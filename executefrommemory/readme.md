# executefrommemory

메모리에 기계어 코드를 직접 써넣고 실행할 수 있는지 테스트한다.


## 배경 지식

CPU가 실행하는 "코드"는 메모리에 올라간 바이트 배열이다.
예를 들어 `return 42;`는 컴파일되면 아래 6바이트가 된다:

```
B8 2A 00 00 00   ->  mov eax, 42   (eax 레지스터에 42를 넣어라)
C3               ->  ret           (호출자에게 돌아가라)
```

CPU는 메모리 주소를 가리키면 거기 있는 바이트를 명령어로 해석하고 실행한다.
즉, 어떤 메모리에든 위의 바이트를 써넣고 CPU가 그 주소로 점프하면 코드가 실행된다.

이를 악용하면 공격자가 데이터 버퍼에 악성 기계어를 넣고 실행할 수 있다.
Windows는 이를 막기 위해 DEP(Data Execution Prevention)를 도입했다.
DEP는 "데이터용 메모리에서는 코드 실행을 금지"하는 보호 기능이다.


## 테스트 흐름

### 1단계: 메모리 할당 + 기계어 코드 복사

`VirtualAlloc`으로 `PAGE_READWRITE`(읽기/쓰기 전용) 메모리를 할당한다.
이 메모리에 `mov eax, 42; ret` 기계어 6바이트를 복사한다.

### 2단계: 실행 권한 없이 호출 -> DEP 차단 확인

할당한 메모리를 함수 포인터로 캐스팅하여 호출한다.
실행 권한이 없으므로 DEP가 Access Violation 예외를 발생시킨다.
`__try/__except`로 예외를 잡아서 프로그램이 죽지 않고 메시지를 출력한다.

### 3단계: 실행 권한 부여 후 호출 -> 정상 실행

`VirtualProtect`로 메모리 보호 속성을 `PAGE_EXECUTE_READ`로 변경한다.
같은 주소를 다시 호출하면 DEP가 차단하지 않고 코드가 실행된다.
리턴값 42가 출력되면 성공이다.


## 실행 결과 예시

```
메모리 할당 성공: 주소 = 000001C2BEF20000

--- 실행 권한 없이 호출 시도 ---
예외 발생! DEP가 실행을 차단했습니다.
-> 읽기/쓰기 전용 메모리에서는 코드를 실행할 수 없습니다.

--- 실행 권한 부여 후 호출 시도 ---
메모리 보호 속성 변경 완료: PAGE_READWRITE -> PAGE_EXECUTE_READ
실행 성공! 리턴값 = 42
-> 메모리에 써넣은 기계어 코드가 정상적으로 실행되었습니다.
```


## 핵심 포인트

- `PAGE_READWRITE` 메모리에서는 코드 실행이 차단된다 (DEP).
- `VirtualProtect`로 `PAGE_EXECUTE_READ`를 부여하면 실행 가능해진다.
- 실제 공격에서는 ROP 기법으로 `VirtualProtect` 호출을 유도하여 DEP를 우회한다.


## 사용하는 Windows API

| API | 역할 |
|-----|------|
| VirtualAlloc | 메모리 할당 |
| VirtualProtect | 메모리 보호 속성 변경 |
| VirtualFree | 메모리 해제 |
